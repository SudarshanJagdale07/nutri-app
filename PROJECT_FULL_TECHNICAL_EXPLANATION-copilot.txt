PROJECT OVERVIEW
This is a comprehensive nutrition tracking application called NutriAI that allows users to log meals via text or image, track daily nutrition intake, get predictive insights, and interact with an AI assistant. The app uses AI (Google Gemini) for parsing food descriptions, analyzing images, and providing personalized nutrition advice. It supports user profiles with calculated BMR, maintenance calories, and daily targets based on personal details like age, weight, height, activity level, and goals (weight loss, muscle gain, or maintenance).

The application consists of a full-stack architecture with a Node.js/Express backend and a React frontend. Users can sign up, set their profile, log meals, view history, get tomorrow's predictions, insights on eating patterns, risk analysis, and chat with an AI for nutrition questions.

TECH STACK
Frontend Frameworks: React 19.2.0 with Vite for build tool, React Router DOM for routing, Zustand for state management.
Backend Frameworks: Node.js with Express 5.2.1, ES modules.
Libraries: Google Generative AI (@google/generative-ai 0.24.1) for LLM text parsing and chat, bcryptjs for password hashing, jsonwebtoken for auth, mongoose for MongoDB ODM, multer for file uploads, cors for cross-origin, dotenv for env vars, AJV for JSON validation.
Databases: MongoDB with native driver and Mongoose for schemas.
APIs: Google Gemini API for AI features, internal REST APIs for frontend-backend communication.
Build Tools: Vite for frontend, npm for package management.
Package Managers: npm for both frontend and backend.
Additional: Tailwind CSS for styling, Framer Motion for animations, Fuse.js for fuzzy search, Recharts for charts, React Hook Form for forms, React Hot Toast for notifications.

Why each is used: React provides a component-based UI; Vite offers fast development; Express handles server routing; MongoDB stores user data, meals, profiles; Gemini enables AI parsing and chat; Zustand manages global state simply.

FILE BY FILE EXPLANATION

File: backend/server.js
Purpose: Entry point of the backend application.
Imports: Express, dotenv, cors, connectDB from config, various route modules.
Functions:
startServer(): Initializes Express app, uses cors and json middleware, mounts routes for auth, profile, api (text food log, meal data, predictive), food-image, api/chat. Loads dotenv, connects DB, listens on PORT (5000 default). Logs presence of API keys.

File: backend/config/db.js
Purpose: Handles MongoDB connection and collection setup.
Imports: MongoClient, mongoose, dotenv.

File: backend/config/index.js
Purpose: Exports connectDB.

File: backend/constants/units.js

APPLICATION STARTUP FLOW
Backend: Run npm start or node server.js. Loads dotenv, calls connectDB to connect MongoDB. Creates Express app, uses cors, express.json. Mounts routes. Listens on PORT, logs API key presence.

Frontend: Run npm run dev. Vite starts dev server. Loads main.jsx, renders App with RouterProvider and Toaster.

REQUEST FLOW (END TO END)
Example: User logs in.
Frontend: Signin form calls auth.js login API, posts email/password to /auth/login.
Backend: authController.login verifies, returns token.
Frontend: Stores token, sets user in store, navigates to dashboard.

User analyzes text meal.
Frontend: FoodTextLogger calls useMealAnalysis.analyzeTextMeal, posts text to /api/analyze-text-meal.
Backend: textFoodLogController.analyzeTextMeal uses LLM or fallback to parse, matches foods, returns meal and candidates.
Frontend: Displays results, allows save.

User saves meal.
Frontend: Calls saveTextMeal, posts meal to /api/save-text-meal.
Backend: Saves to meals, updates daily_nutrition, tracks estimated.

API STRUCTURE
POST /auth/register: Register user. Body: {name, email, password}. Response: {token, user}.
POST /auth/login: Login. Body: {email, password}. Response: {token, user}.

POST /profile/update: Update profile. Body: profile data. Response: profile.
GET /profile/:userId: Get profile. Response: profile.

POST /api/analyze-text-meal: Analyze text. Body: {text, userId}. Response: {meal, candidates}.
POST /api/save-text-meal: Save meal. Body: {meal, userId}. Response: {meal}.

POST /food-image/analyze: Analyze image. Multipart image. Response: {success, ml, preview/analysis}.
POST /food-image/log-image-meal: Log image meal. Body: {userId, items, totals}. Response: {meal}.

GET /api/meals/:userId: Get meals. Query: limit, before. Response: {meals}.
POST /api/daily/add: Add to daily. Body: {userId, date, totals, mealId}. Response: {daily}.
GET /api/daily/:userId/:date: Get daily. Response: {daily}.
GET /api/food/:name: Lookup food. Response: food doc.

GET /api/predictive/:userId/tomorrow: Tomorrow prediction. Response: prediction data.
GET /api/predictive/:userId/insights: Insights. Response: insights.
GET /api/predictive/:userId/risk: Risk analysis. Response: risk data.

POST /api/chat: Chat. Body: {userId, message, history}. Response: {reply, context}.

DATA FLOW
Frontend state: Zustand store for user. Hooks manage API data.
API calls: Fetch profile on dashboard load. Analyze meal on input. Save on confirm.
JSON data: Requests/responses in JSON. Nutrition values as numbers.
UI updates: After API, update local state, re-render components.

FRONTEND WORKFLOW
Routing: React Router with protected routes.
Components: Dashboard integrates logging, stats. FoodTextLogger handles text input, analysis, save. Calls analyzeTextMeal API, displays meal, candidates. On save, calls saveTextMeal, refreshes daily totals.
State management: Zustand for user. Hooks for data.
API communication: apiManager functions use fetch to backend.
UI updates: After save, refresh daily nutrition hook, update rings.

BACKEND WORKFLOW
Routes: Mounted in server.js, handle requests to controllers.
Controllers: Auth handles auth. TextFoodLog analyzes/saves. ImageFoodLog for images. MealData for data. Predictive for insights.
Services: BMR computes targets. FoodMatching finds docs. LLM for AI. Persistence saves data.
Middleware: None specific, uses cors, json.

DATABASE STRUCTURE
Type: MongoDB.
Collections: users (User schema), userprofiles (UserProfile), meals (Meal), daily_nutrition (DailyNutrition), food_nutrition_DB (nutrition items), llm_estimated_foods (estimated).
Relationships: Meals reference User. DailyNutrition aggregates per user/date. Profiles reference User.

IMPORTANT FUNCTIONS
computeServerDerived: Calculates all nutrition targets from profile.
findNutritionDocByName: Robust food matching with scoring.
analyzeTextMeal: Parses text, matches, estimates.
saveMeal: Inserts meal document.
updateDailyNutrition: Increments daily totals.
getPredictiveTomorrow: Predicts based on history.
handleChat: Chats with Gemini using user context.

ERROR HANDLING
Backend: Try-catch in controllers, return 400/500 with messages. LLM fallbacks to simpleParse. ML errors logged.
Frontend: Toast notifications for errors. Loading states.

SECURITY FEATURES
Auth: JWT tokens, bcrypt hashing. Protected routes check token. Profile updates require userId.

COMPLETE EXECUTION FLOW SUMMARY
User opens app: Loads home page. Signs up: Registers, sets profile. Logs in: Gets token, dashboard loads profile, daily totals. Logs meal: Analyzes text/image, shows preview, saves to DB, updates daily. Views predictions: Fetches tomorrow/insights/risk. Chats: Sends message, gets AI reply with data context.
